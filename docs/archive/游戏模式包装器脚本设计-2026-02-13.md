# 游戏模式包装器脚本设计（每游戏包装器 + 状态机 + 容错恢复）

## 1. 目标
为单个游戏（先以《赛博朋克 2077》为例）提供可控、可回滚的“游戏模式”自动化：
- 启动前自动切换到稳定显示配置（例如 119.88Hz + VRR 策略）
- 游戏退出后自动恢复桌面配置（例如 144Hz）
- 异常退出/崩溃时仍可恢复（避免系统卡在错误配置）

## 2. 为什么用“每游戏包装器”而不是全局侦测
优点：
1. 触发更准确：只在该游戏启动时执行，避免误判其他程序。
2. 变量更少：每个游戏可以有独立参数和恢复逻辑。
3. 调试简单：日志明确、回滚路径清晰。
4. 风险更低：不会因为后台误侦测导致全局显示配置来回跳。

## 3. 架构分层

### 3.1 配置层（Config）
每个游戏一个配置块：
- 游戏名 / AppID（如 1091500）
- 启动前显示配置（mode, VRR）
- 退出后恢复配置
- 启动命令模板（`%command% --launcher-skip`）
- 超时与重试策略

建议位置：
- `~/.config/game-mode-wrapper/config.env` 或 `config.toml`

### 3.2 执行层（Runner）
负责：
1. 加载配置
2. 进入状态机
3. 调用系统命令（`kscreen-doctor` / `kwriteconfig6` / 游戏命令）
4. 记录日志

建议脚本：
- `~/bin/game-mode-wrapper.sh`

### 3.3 状态机层（State Machine）
建议状态：
1. `IDLE`：空闲
2. `PREPARE`：保存当前显示状态快照
3. `APPLY_GAME_PROFILE`：应用游戏显示配置
4. `LAUNCH_GAME`：启动游戏并等待
5. `MONITOR`：监测进程生命周期
6. `RESTORE_DESKTOP_PROFILE`：恢复桌面配置
7. `DONE`：完成
8. `ERROR_RECOVERY`：异常恢复分支

状态机约束：
- 每个状态只做一件事
- 每步都要有返回码检查
- 失败立即进入 `ERROR_RECOVERY`

### 3.4 容错恢复层（Recovery）
必须覆盖以下场景：
1. 游戏崩溃
2. Steam 中断
3. 脚本被中断（Ctrl+C / 会话异常）
4. 应用配置失败

策略：
- `trap` 捕获 `EXIT INT TERM`，统一执行恢复函数
- 使用 `lockfile` 防止并发启动
- 使用“上次稳定配置快照”兜底恢复

## 4. 关键设计点

### 4.1 输出设备动态解析
不要硬编码 `output.1`。
- 启动时通过 `kscreen-doctor -o` 解析当前活动输出（如 HDMI-A-1 对应 id）
- 运行中引用动态 id

### 4.2 幂等与防抖
- 若当前已是 119.88Hz，不重复切换
- 避免短时间内重复 apply/restore 引发闪屏

### 4.3 日志规范
建议日志文件：
- `~/.local/state/game-mode-wrapper.log`

记录内容：
- 时间戳
- 当前状态
- 执行命令
- 返回码
- 恢复动作是否成功

### 4.4 可观测性
- 提供 `--dry-run` 模式（只打印不执行）
- 提供 `status` 子命令查看当前锁与状态

## 5. Steam 集成方式
在游戏启动项中调用包装器，而不是直接写复杂环境变量：

示例：
`~/bin/game-mode-wrapper.sh --game cp2077 -- %command% --launcher-skip`

包装器内部再决定是否附加：
- `PROTON_ENABLE_WAYLAND=1`
- `PROTON_ENABLE_HDR=1`
- 其他实验参数

这样 Steam 启动项保持简洁，可维护性更高。

## 6. 最小可行实现（MVP）
MVP 只做 4 件事：
1. 保存当前模式（快照）
2. 切换到 119.88Hz
3. 启动并等待游戏退出
4. 恢复原模式

后续再增加：
- VRR 策略切换
- HDR 模式策略
- 多游戏配置
- 智能家居联动

## 7. 风险与边界
1. KDE/驱动升级后，`kscreen-doctor` 字段可能变化（需要解析容错）
2. 多显示器环境下恢复复杂度更高（MVP 先单屏）
3. 某些设置改动可能需要短暂重建显示链路（可接受）

## 8. 建议的落地顺序
1. 先完成 cp2077 单游戏 MVP
2. 连续运行 10 次（含异常中断）验证恢复是否可靠
3. 再扩展成可复用模板

## 9. 阶段性结论
“每游戏包装器脚本 + 状态机 + 容错恢复”是当前环境下最务实、最稳定、可长期维护的自动化路线。
